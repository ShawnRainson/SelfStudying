🧠 Теория дня
Тема: "Почему код пишут не только для компьютера"

Самая большая ошибка новичков — думать, что задача программиста написать код, который работает.

Нет.

Задача программиста — написать код, который:

работает;
понятен;
легко изменяется;
легко тестируется;
который через полгода поймёт другой человек.

Есть известная цитата:

"Programs must be written for people to read, and only incidentally for machines to execute."

Запомни её. Она станет одним из наших принципов.

Хороший код

Например:

total_price = quantity * price

Даже без комментариев понятно, что происходит.

Плохой вариант:

x = a * b

Компьютеру всё равно.

Разработчику — нет.

Первое правило нашей команды

Каждое имя должно объяснять своё назначение.

Если переменную приходится комментировать — скорее всего, она плохо названа.

🧠 Теория
Что такое функция?

Представь кофейню.

Клиент говорит:

Сделайте мне латте.

Бариста не придумывает рецепт заново.

Он выполняет уже известную последовательность действий.

То же самое делает функция.

Вместо того чтобы писать один и тот же код несколько раз, мы даём ему имя и вызываем тогда, когда он нужен.

Плохой код
print("Hello Alex")
print("Hello John")
print("Hello Kate")

Если завтра формат приветствия изменится, придётся исправлять каждую строку.

Хороший код
def say_hello(name):
    print(f"Hello, {name}")

Использование:

say_hello("Alex")
say_hello("John")
say_hello("Kate")

Изменить нужно только одну функцию.

Почему это важно?

Запомни правило:

Каждая функция должна делать одну понятную вещь.

Не "обработать всю программу".

А:

добавить книгу;
удалить фильм;
показать меню;
найти пользователя.

Это называется принцип единственной ответственности. Мы ещё не раз к нему вернёмся.

Мы уже использовали списки, словари и массивы. Сегодня начнём понимать, почему выбирается одна структура данных, а не другая.

🧠 Теория дня

Представь, что тебе нужно хранить информацию о людях.

Есть три варианта.

Вариант 1. Список
users = [
    "Alex",
    "Kate",
    "John"
]

Хорошо?

Да.

Но где возраст?

Где город?

Где телефон?

Вариант 2. Список списков
users = [
    ["Alex", 20],
    ["Kate", 18]
]

Работает.

Но...

Что означает

users[0][1]

?

Через неделю уже забудешь.

Вариант 3. Словарь
user = {
    "name": "Alex",
    "age": 20
}

Теперь

user["age"]

понятно даже человеку, который впервые увидел этот код.

Первый принцип выбора структуры данных

Спроси себя:

Как я буду искать информацию?

Если по номеру — список.

Если по имени (ключу) — словарь.

Если нужны уникальные значения — множество (set).

Эту привычку мы будем развивать постоянно.

Теория

Мы вчера передавали список в метод.

Представь:

Есть коробка с книгами.

Можно:

❌ сделать новую коробку;

или

✅ дать человеку ссылку на существующую коробку.

Когда ты пишешь:

ShowBooks(books);

ты передаёшь ссылку на список, а не копируешь все элементы. Это эффективнее и позволяет методу работать с теми же данными.

🧠 Теория

Представь фильм.

У него есть:

название
год
рейтинг

Можно хранить так:

["Avatar", 2009, 9.2]

Работает.

Но что означает

movie[2]

?

Через месяц уже забудешь.

Поэтому существуют объекты.

Python

movie = {
    "title": "Avatar",
    "year": 2009,
    "rating": 9.2
}

JavaScript

const movie = {
    title: "Avatar",
    year: 2009,
    rating: 9.2
};

Видишь?

Практически одинаково.

Как получить данные?

Python

movie["title"]

JavaScript

movie.title

или

movie["title"]

Оба варианта правильные.

Почему это лучше списка?

Потому что

movie.rating

понятнее, чем

movie[2]

💜 C#
Теория

Есть два похожих понятия.

Метод

Что-то делает.

Параметр

То, что получает метод.

Например

ShowBooks(books)

books внутри объявления метода — параметр.

А конкретный список, который ты передаёшь при вызове, — аргумент.

Эти слова часто путают, поэтому важно различать их.

Практика

Добавь метод

UpdateBook(List<string> books)

Алгоритм:

Ввести старое название.

Если книга есть —

ввести новое.

Заменить.

Вывести список.

⚠ Не используй индексатор списка напрямую.

Попробуй решить через цикл.
